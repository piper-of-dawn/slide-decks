Slide 01 / Current process

Current Build Flow: Single Powerful GCP VM

A Jenkins job starts one large build machine, runs the C++ build through agent.jar, then shuts the VM down after completion.

Bottleneck
  • Only one build VM exists.
  • Multiple users create a queue.
  • Parallelism is limited by VM count.
  • Startup time is paid repeatedly.
5-second read

The system is elastic in time, not in capacity: every build waits for the same single worker path.

Slide 02 / Target operating model

Planned Split: Terraform for Foundation, GCP Plugin for Elastic Agents

Keep stable infrastructure declarative. Let Jenkins request short-lived builders only when queued work needs capacity.

Layer Responsibility
TerraformPermanent foundation Manages Jenkins controller VM, service accounts, IAM permissions, firewall rules, networks, disks, buckets, backups, and other stable infrastructure.
GCP Jenkins PluginElastic build agents Creates temporary VMs when jobs are queued, assigns agents by labels such as cpp-builder, runs the C++ build, keeps agents warm for idle timeout, then deletes them.
Scaling ruleBuild capacity Parallel builds scale up to the configured maximum instance limit instead of waiting behind one permanent builder.

Terraform should not create every short-lived build VM. Terraform creates the base infrastructure; the GCP Jenkins plugin handles elastic build capacity.
